# 检查工具三项改进 — 设计文档

> 日期：2026-07-01 | 状态：待实现

## 概述

三个改进点：

1. **草稿系统升级** — 上限从 3 条扩至 6 条，支持删除
2. **导入文件** — 支持导入 .docx 报告和照片 OCR，合并到已有草稿
3. **AI 优化可控** — 可中断请求，可不选 AI 结果直接用原文

---

## 改进一：草稿系统升级

### 现状问题

IndexedDB 用 `reportType`（safety/5s/company）作为 keyPath → 每种类型只能存 1 条草稿，总共最多 3 条。无法删除。

### 设计方案

**存储改造：**

- 更换 keyPath 为 `id`（UUID，如 `draft_1719000000_a1b2c3`）
- 保存时生成唯一 ID，不再按类型覆盖
- 列表按 `updatedAt` 降序排列
- 新草稿超过 6 条时，自动删除最旧的一条
- `deleteDraft(id)` 已有函数，只需改为传 ID

**UI 改造：**

- 首页草稿区标题从「未完成的草稿」→「草稿箱 (N/6)」
- 每条草稿卡片：
  - 左侧：类型标签（安全=蓝 / 5S=橙 / 公司=绿），后跟条目数和日期
  - 右侧：🗑️ 删除按钮
- 点击删除 → confirm 确认 → 删除并刷新列表
- 点击卡片主体 → 继续编辑（现有逻辑不变）

### 涉及文件

| 文件 | 改动 |
|------|------|
| `js/db.js` | store keyPath 改为 id；saveDraft 生成 ID + 限制 6 条 |
| `js/ui.js` — `renderHomePage` | 草稿卡片改造：类型标签、删除按钮、容量指示 |
| `js/app.js` — `showHome` / `showItemList` | 适配新的 draft 数据结构（data.type 字段） |

## 改进二：导入文件

### 设计方案

**入口：首页**

在三种报告类型卡片上方，增加虚线边框的导入按钮：

```
📥 导入报告文件
支持 Word 报告 (.docx) 或照片
```

点击后弹出底部选择面板。

**导入流程（方案 A — 询问合并）：**

```
选择文件 → 解析 → 预览解析结果 → 选择合并目标 → 确认导入 → 跳转编辑
```

合并目标选择：
1. **合并到最近编辑的草稿**（默认选中，覆盖 90% 场景）
2. 合并到其他草稿（列出所有草稿供选择）
3. 创建新草稿

### Word 导入 (.docx)

**技术方案：**

1. 用户选择 `.docx` 文件 → `FileReader` 读取为 ArrayBuffer
2. 用 JSZip（CDN 加载，约 100KB）解压
3. 读取 `word/document.xml`，解析 `<w:tbl>` 表格
4. 逐行提取：
   - 问题描述（问题描述/存在问题列的文字）
   - 整改前图片（`<w:drawing>` → `r:embed` → `word/media/imageN.jpeg`）
   - 整改后图片（同上）
5. 构造 items 数组，生成新草稿或合并

**约束：**
- 只解析本工具生成的格式（表格结构已知）
- 非本工具格式的 .docx 可能解析失败，给出友好提示
- 提取的图片直接 base64 存入，不额外下载

### 照片 OCR

**技术方案：**

1. 用户拍照或选照片
2. Canvas 压缩到 < 1MB
3. 调用豆包视觉模型（需确认 API 可用性）
4. Prompt：「请识别这张安全检查照片中的问题，用简洁的整改报告语言描述。只输出问题描述，不要其他内容。」
5. 返回的描述自动填入新条目的问题描述框
6. 照片自动作为整改前照片

**约束：**
- 需要豆包视觉模型 endpoint（可能需要单独申请）
- OCR 结果标注「AI 识别，请核对修改」
- 如果视觉模型不可用，照片仅作为图片导入，描述留空

### 涉及文件

| 文件 | 改动 |
|------|------|
| `js/importer.js` | **新建** — Word 解析、照片 OCR、合并逻辑 |
| `js/ui.js` — `renderHomePage` | 导入按钮 + 导入选择面板 |
| `js/app.js` | 导入流程协调 |
| `index.html` | 引入 JSZip CDN |
| `lib/jszip.min.js` | 可能需要本地化（考虑国内 CDN） |

## 改进三：AI 优化可控

### 现状问题

- AI 请求发出后无法取消，只能等待或关页面
- 结果页必须从 3 个选项中选一个，不能直接用原文

### 设计方案

**AI 等待中 — 增加取消按钮：**

- 在加载动画下方增加 `✕ 取消` 按钮
- 点击 → 调用 `AbortController.abort()` → 终止 fetch → 返回编辑页
- 不提示错误，静默返回

**技术实现：**

```js
// ai.js — callDoubaoOptimize 增加 signal 参数
async function callDoubaoOptimize(text, reportType, signal) {
  const response = await fetch(DOUBAO_API_URL, {
    // ... 其他参数
    signal  // AbortController.signal
  });
  // ...
}
```

**结果页 — 增加「使用原文」按钮：**

- 在「换一批」按钮下方增加 `📋 直接使用原文（不用 AI 结果）` 按钮
- 点击 → 行为等同于选了原文（`onSelect(text)` 用原始描述）
- 样式：灰边框白底，不要抢眼但要看得见

### 涉及文件

| 文件 | 改动 |
|------|------|
| `js/ai.js` — `callDoubaoOptimize` | 增加 `signal` 参数 |
| `js/app.js` — `showOptimizePage` | 创建 AbortController，传 signal，处理取消 |
| `js/ui.js` — `renderOptimizePage` | 加载态加取消按钮；结果页加使用原文按钮 |

## 数据库迁移

IndexedDB keyPath 从 `type` 改为 `id` 是 breaking change。

**迁移策略：**
1. 尝试用旧格式读取所有草稿
2. 为每条生成 ID，写入新格式
3. 删除旧格式数据
4. 如果旧格式读取失败（说明已是新格式），跳过迁移

`db.js` 中 bump `DB_VERSION` 到 2，在 `onupgradeneeded` 中处理迁移。

## 实施顺序

建议按以下顺序分步实施（每步一个 commit）：

1. **草稿系统升级**（基础改造，其他功能依赖它）
2. **AI 优化可控**（独立改动，不影响其他）
3. **导入 .docx**（先做 Word，因为不需要额外 API）
4. **导入照片 OCR**（依赖豆包视觉模型，可能需额外调试）
